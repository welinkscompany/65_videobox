# VideoBox 오픈소스 리서치 및 범위 축소 문서

## 1. 목적

이 문서는 VideoBox 구현 전에 다음을 확정하기 위한 문서다.

- 어떤 기술과 오픈소스를 실제로 채택할지
- 어떤 것은 지금 넣지 않을지
- 대시보드/앱은 어떤 언어와 프레임워크로 갈지
- BrollBox에서 무엇을 가져올지
- 최종적으로 첫 구현 범위를 어디까지 줄일지

핵심 원칙:

- 좋아 보이는 기술보다 `지금 바로 구현 가능한 기술`을 우선한다
- 기능이 많아지는 것보다 `첫 결과물이 빨리 나오는 구조`를 우선한다
- 완성형 제품 환상보다 `검증 가능한 MVP`를 우선한다

## 3. 핵심 관찰

### 3.1 Reddit 관찰

실제 자동화 사용자들이 반복해서 말하는 공통점은 다음과 같다.

- 완전 자동 최종본보다 `반복 구조를 가진 영상`이 자동화에 잘 맞는다
- 자막/스크립트 기반 편집이 가장 자동화하기 쉽다
- B-roll을 쓰려면 `설명 가능한 메타데이터`가 먼저 있어야 한다
- 전사 품질이 높아도 마지막 수동 검수는 여전히 필요하다

이건 VideoBox 방향과 잘 맞는다.

즉, 첫 구현은:

- 설명형/나레이션 기반
- 반복적인 영상 구조
- 자동 초안 생성 + 사람 검수

로 좁혀야 한다.

### 3.2 Hugging Face / 모델 관찰

- Whisper 계열은 전사 정확도와 범용성 면에서 여전히 강력하다
- CLIP 계열은 이미지/텍스트 매칭에 유용하지만, 바로 제품 품질을 보장하지는 않는다
- sentence-transformers 계열은 문서/스크립트 의미 검색에 구현 부담이 낮다

즉:

- 전사는 `WhisperX`
- 텍스트 의미 검색은 `sentence-transformers`
- 멀티모달 검색은 `open_clip`

이 가장 자연스럽다.

하지만 CLIP 계열은 초기에 과투자할 가능성이 있으므로 1차 MVP에서는 optional로 둬야 한다.

### 3.3 BrollBox 관찰

BrollBox는 VideoBox의 전신으로서 유효하다.

특히 재사용 가치가 높은 것은:

- CapCut export
- auto cut
- STT/정렬 아이디어
- script matching 구조
- shorts 추출 흐름

반면 그대로 가져오면 안 되는 것은:

- Streamlit UI
- Google Sheets/Drive 중심 구조
- provider 하드코딩

## 4. 대시보드/앱 기술 선택

## 결론

### 대시보드 언어

- `TypeScript`

### 대시보드 프레임워크

- `React + Vite`

### 로컬 API / 코어 엔진 언어

- `Python`

### 로컬 API 프레임워크

- `FastAPI`

## 이유

이 조합이 가장 현실적이다.

- 대시보드는 React + TypeScript가 가장 빠르게 만들 수 있다
- FastAPI는 Python 코어 엔진과 붙이기 쉽다
- FFmpeg, WhisperX, 자산 인덱싱, CapCut export는 Python 쪽이 유리하다

## 중요한 범위 축소 결정

처음부터 `Tauri`까지 바로 들어가지 않는다.

이유:

- 지금 핵심은 패키징이 아니라 편집 파이프라인 검증이다
- 대시보드를 웹 UI로 먼저 띄우면 개발 속도가 빠르다
- 나중에 안정화 후 Tauri로 감싸는 것이 더 안전하다

따라서:

- `v1 UI = React + TypeScript 웹 대시보드`
- `v1.5 이후 = 필요하면 Tauri 데스크톱 래핑`

이 방식으로 간다.

## 5. 채택 후보 분류

### 5.1 바로 채택

#### A. FastAPI

역할:

- 로컬 API
- job orchestration entrypoint
- 프론트엔드와 엔진 연결

판단:

- `채택`

이유:

- Python 기반 엔진과 잘 맞는다
- 구현 속도가 빠르다
- 나중에 원격 API로도 확장 가능하다

#### B. WhisperX

역할:

- STT
- 단어 단위 타임코드
- 정렬

판단:

- `채택`

이유:

- VideoBox의 핵심인 자막/컷 정렬 문제와 직접 연결된다
- BrollBox에서도 같은 계열 접근이 이미 유효했다

주의:

- diarization/겹침 발화/엣지 케이스는 완벽하지 않다
- review flag 구조가 필수다

#### C. PySceneDetect

역할:

- 긴 원본 영상 샷 분리
- rough cut 전처리

판단:

- `채택`

이유:

- 긴 원본을 정리하는 핵심 보조 도구다
- 직접 구현보다 빠르게 검증할 수 있다

주의:

- 진짜 이야기 단위 scene이 아니라 shot detection에 가깝다
- 장면 의미 분리는 별도 계층에서 처리해야 한다

#### D. sentence-transformers

역할:

- 문서/스크립트 의미 검색
- 세그먼트 유사도 매칭
- B-roll 추천의 텍스트 쪽 랭킹 보조

판단:

- `채택`

이유:

- 구현 부담이 낮고 실효성이 높다
- 초기에는 CLIP보다 바로 제품에 붙이기 쉽다

#### E. BrollBox CapCut export 로직

역할:

- CapCut draft export

판단:

- `채택`

이유:

- 새로 0부터 만드는 것보다 훨씬 빠르다
- 이미 내부적으로 실험된 흐름이다

주의:

- VideoBox timeline schema 기준으로 adapter 재구성 필요

#### F. BrollBox auto cut 로직

역할:

- 긴 원본 영상 분할
- 어두운 구간/정지 구간 제거

판단:

- `채택`

이유:

- 바로 재사용 가치가 높다

### 5.2 검증 후 채택

#### G. open_clip

역할:

- 이미지/프레임과 텍스트의 멀티모달 검색
- B-roll 자동 추천 강화

판단:

- `검증 후 채택`

이유:

- 장기적으로 매우 유용하다
- 하지만 첫 구현에서 반드시 필요한 건 아니다

채택 조건:

- sentence-transformers 기반 텍스트 검색만으로 추천 품질이 부족할 때
- B-roll 라이브러리 규모가 커져서 프레임 기반 검색이 필요할 때

#### H. Voicebox

역할:

- 로컬 TTS
- 음성 클로닝
- 음성 입력/출력 실험

판단:

- `제한 범위 채택`

이유:

- `jamiepine/voicebox`는 로컬 우선 음성 스튜디오로서 방향이 좋다
- Windows/macOS/Linux를 지원하고, 여러 TTS 엔진과 로컬 워크플로우를 제공하는 점은 매력적이다
- 특히 나중에 VideoBox가 `나레이션 재생성`, `가이드 음성`, `TTS 기반 설명 영상`까지 확장될 경우 참고 가치가 크다

하지만 채택 범위를 아주 좁혀야 한다.

- 첫 구현의 핵심은 `본인 목소리 기반 나레이션 대체`이지 `음성 합성 스튜디오`가 아니다
- voice cloning은 품질/윤리/동의 관리 리스크가 있다
- upstream 이슈를 보면 로컬 API 보안, 모델 품질 편차, 엔진별 안정성 이슈가 아직 있다
- 잘못 넣으면 편집 엔진보다 음성 생성 기능이 범위를 키울 가능성이 높다

따라서 채택 범위는 다음으로 제한한다.

- 사용자 본인 목소리만 사용
- 나레이션 대체용 TTS만 지원
- 잘못 발음했거나 다시 녹음하기 번거로운 일부 구간 대체만 목표
- 범용 음성 클로닝 툴, 다중 화자 관리, 퍼블릭 음성 마켓 같은 방향은 제외

권장 활용 방식:

- `TTSProvider` 뒤에 제한적 기능으로 붙인다
- 편집 파이프라인 안에서 `구간 대체` 용도로만 사용한다
- 자동 전면 대체가 아니라 review 기반 적용으로 간다

#### I. Remotion

역할:

- 설명형 오버레이
- 카드/표/숫자 강조
- 간단한 모션 그래픽

판단:

- `검증 후 채택`

이유:

- 설명형 비주얼 제작에 잘 맞는다
- 하지만 첫 구현에서 바로 넣으면 범위가 커진다

채택 조건:

- preview 렌더와 CapCut export 흐름이 먼저 안정화된 뒤
- 설명형 카드/오버레이가 실제 병목으로 확인될 때

#### J. Tauri

역할:

- 로컬 데스크톱 앱 패키징

판단:

- `검증 후 채택`

이유:

- 장기적으로 맞는 방향이지만
- 지금은 앱 패키징보다 편집 엔진 검증이 더 중요하다

채택 조건:

- 웹 대시보드 기반 로컬 워크플로우가 안정화된 뒤

### 5.3 지금은 제외

#### K. Streamlit

판단:

- `제외`

이유:

- 빠른 프로토타입엔 좋지만 장기 구조와 안 맞는다

#### L. Electron

판단:

- `제외`

#### M. 풀 자체 편집기

- `제외`

이유:

- VideoBox의 첫 가치가 아니다
- 타임라인 편집기 개발은 별도 제품 난이도다

#### N. 생성형 이미지/비디오 자동 제작

판단:

- `제외`

이유:

- 품질 일관성, 라이선스, 환각 리스크가 크다
- 첫 구현은 `추천/조합형`에 집중해야 한다

## 6. BrollBox에서 실제로 가져올 것

### 즉시 가져올 것

- CapCut export 구조
- auto cut 기준값과 처리 흐름

### 구조만 참고할 것

- transcribe_audio의 정렬 개념
- match_script의 장면 분리/키워드 추출 흐름
- shorts_clip의 후보 추출 흐름
- Voicebox의 로컬 음성 스튜디오/엔진 추상화 방향

### 가져오지 않을 것

- Streamlit UI 전체
- Google Sheets/Drive 의존 구조
- 하드코딩된 Gemini 중심 구조

## 7. 첫 구현 범위 최종 축소안

## 최종 결정

첫 구현은 아래만 만든다.

1. 로컬 프로젝트 생성
2. 녹음 파일 입력
3. 참고 문서/스크립트 입력
4. 사용자 본인 목소리 샘플 입력
5. B-roll 메타데이터 인덱싱
6. WhisperX 기반 전사
7. 세그먼트 분리
8. 반복 take / 무음 / 재시작 탐지
9. 제한적 TTS 대체 생성
10. sentence-transformers 기반 텍스트 추천
11. 간단한 B-roll 추천
12. 기본 음악 추천
13. timeline JSON 생성
14. preview 또는 CapCut export
15. 기본 review 대시보드

## 첫 구현에서 빼는 것

1. 긴 원본 영상 전체 자동 이해
2. open_clip 기반 멀티모달 추천
3. 범용 음성 클로닝 스튜디오 기능
4. Remotion 기반 고급 설명형 애니메이션
5. 숏폼 자동 생성 고도화
6. Tauri 패키징
7. SaaS 인증/클라우드 기능

## 10. 결론

VideoBox는 지금 너무 많은 것을 동시에 하려 하면 느려진다.

따라서 첫 구현은 다음 한 문장으로 고정한다.

`나레이션 오디오와 참고 문서를 받아 전사, 세그먼트 분리, 본인 목소리 기반 제한적 TTS 대체, B-roll 추천, 기본 음악 추천, CapCut export 초안을 만드는 로컬 편집 엔진과 웹 대시보드`

이 범위까지만 만들면 된다.

그 이후에만:

- 멀티모달 검색
- 고급 설명형 비주얼
- 데스크톱 패키징
- SaaS 확장

으로 확장한다.
