# 루미 대시보드 사용자 문구 설계

**작성일:** 2026-07-17  
**상태:** 사용자 승인 설계, 구현 전 검토 대기

## 1. 목표

내부 대시보드에서 사용자가 실제로 조작하는 모든 화면의 시스템 중심 설명을 제작자 중심의 짧은 안내로 바꾼다. 영상 도우미의 이름은 **루미**이며, 도움이 필요한 순간에만 1인칭으로 안내한다.

사용자는 내부 구현 방식, API, provider, runtime, proposal lifecycle을 알 필요가 없다. 각 문구는 사용자가 지금 할 수 있는 행동과 그 결과를 알려야 한다.

## 2. 문체와 표기 원칙

- 기본 제목·설명은 중립적이고 짧게 쓴다. 예: `영상, 음악, 자막을 한곳에서 편집하세요.`
- 추천, 진행, 회복, 실패처럼 도움을 주는 순간에는 루미를 주어로 쓴다. 예: `루미가 장면에 맞는 영상을 골라봤어요.`
- 한 문장에는 한 행동 또는 한 결과만 담는다. 기술적인 이유를 사용자에게 설명하지 않는다.
- 버튼은 동사 중심으로 쓴다. 예: `추천 보기`, `프로젝트에 추가`, `변경 적용`, `다시 확인`.
- 오류 상태에도 다음 행동을 남긴다. 예: `루미가 지금 추천을 만들 수 없어요. 직접 골라 계속 편집할 수 있어요.`
- 코드명, 영문 내부 용어, API·모델·provider 이름은 보이지 않게 한다. 단, 제작 결과의 실제 형식명(예: MP4, SRT, CapCut)은 필요한 곳에서 유지한다.
- `루미`는 사람이 아닌 도우미로 표현한다. 감정적 사과, 능력 과장, 외부 서비스 호출 암시는 사용하지 않는다.

## 3. 용어 사전

| 내부/기존 표현 | 사용자 표현 | 적용 원칙 |
| --- | --- | --- |
| Local Media Director / Director | 루미 / 영상 도우미 | 화면 제목은 `루미`, 첫 소개나 접근성 이름에는 `루미 영상 도우미` |
| proposal / candidate | 루미의 추천 / 추천 항목 | ID, revision은 숨긴다 |
| preflight | 적용 전 확인 | 정상일 때는 가능한 한 표시하지 않고 문제 때만 안내 |
| revision / session | 현재 편집본 | 식별자와 버전 번호는 숨긴다 |
| materialize | 프로젝트에 추가 | 실제 파일 이동·복사 방식은 언급하지 않는다 |
| runtime / capability / provider | 루미 준비 상태 | Settings에서는 사용 가능한 기능과 확인 방법만 보여 준다 |
| pipeline | 제작 흐름 | 화면 구획에만 사용한다 |
| blocked / error | 추천을 만들 수 없음 / 문제가 생김 | 수동 편집 지속 행동을 함께 제시한다 |

## 4. 화면별 적용 범위

### 편집 화면과 루미 패널

- 패널 제목, 열기/닫기 제어, 상태, 입력 label, 추천 비교, 적용 범위, 빈 화면, 재시도·오류·사전 확인 문구를 사용자 말로 바꾼다.
- 번호 참조는 사용자가 고를 수 있도록 `추천 영상 3번`처럼 유지하되, immutable ID·revision·raw diff JSON은 화면에 노출하지 않는다.
- 추천 생성은 `루미에게 추천받기`, 갱신은 `추천 다시 만들기`, 적용은 `이 추천 적용`처럼 결과가 분명한 행동으로 쓴다.

### 수동 미디어 라이브러리

- `Director blocked/error`와 session mutation 같은 문구를 제거한다.
- 루미의 추천이 실패해도 수동 편집을 계속할 수 있음을 자연어로 안내한다.
- 미리보기와 프로젝트 반영의 차이를 `미리보기는 편집본을 바꾸지 않아요. 추가한 뒤 적용하세요.`처럼 설명한다.

### 설정 및 준비 상태

- `runtime`, `loopback`, `fallback`, `API key` 등 구현 설명을 노출하지 않는다.
- 사용자가 취할 행동을 `루미가 이 컴퓨터에서 사용할 수 있는 기능을 확인합니다.`로 표현하고, 준비 실패 시 재확인 방법을 제공한다.
- 외부 자동 호출을 하지 않는다는 제품 약속은 필요할 때 `루미는 이 컴퓨터에서 준비된 기능만 사용해요.`로 짧게 표시한다.

### 프로젝트·자산·제작 결과 화면

- 빈 상태, 진행, 완료, 실패, 내보내기, 자막, 검수 안내를 같은 원칙으로 전수 점검한다.
- 기술 데이터가 제작 판단에 직접 필요한 경우에만 별도 `세부 정보` 영역으로 남긴다. 일반 흐름의 설명문에는 포함하지 않는다.

## 5. 범위 제외와 안전장치

- API 계약, backend 상태 코드, 저장 데이터, 자동화/검증 용어, 개발자 전용 로그는 바꾸지 않는다.
- 접근성 label은 사용자 의미를 유지하며, 로케이터용 `data-*` 속성은 바꾸지 않는다.
- 오류의 원인을 숨겨 진단 가능성을 해치지 않도록, 개발자용 원문은 console/test fixture에만 유지하고 사용자 화면은 행동 가능한 요약으로 바꾼다.
- 전체 App 구조를 재작성하지 않고, 화면에 렌더링되는 copy와 그것을 검증하는 테스트만 바꾼다.

## 6. 검증 기준

- 사용자 화면의 JSX/컴포넌트 문자열을 전수 인벤토리하고, 내부 용어가 일반 사용자용 화면에 남지 않았는지 검색으로 검증한다.
- 대표 흐름(프로젝트 생성, 편집, 루미 추천/적용, 수동 라이브러리, 분석/준비 상태, 내보내기, 오류·빈 상태)의 UI 테스트를 RED→GREEN으로 갱신한다.
- 접근성 이름과 버튼 동작은 유지하고 focused frontend suite, full frontend suite, production build를 통과한다.
- 독립 사양 리뷰와 문구/코드 품질 리뷰에서 범위 누락과 기술 정보 노출을 확인한다.
